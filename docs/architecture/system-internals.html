System Internals How each installed piece of Temperance Engine actually works, mechanically

Scope. This doc is the mechanics reference: exact flags, env vars, file writes, endpoints, and fallback behavior for every script and service in this repo. For the productization narrative and business context, see architecture.html. For how these pieces connect to each other (and where they don't), see integration-map.html.

1. Install Orchestration State Machine

install.sh is a pure flag → env-var → script-dispatch pipeline. It never installs anything itself — every write happens in a downstream scripts/install-*.sh.

Each var is exported as TEMPERANCE_<X>_MODE, then the following seven scripts run in this exact, fixed order — unconditionally invoked (each script decides internally whether to act, based on its own mode var):

install-pai.sh            # AGENTS.md always; CLAUDE.md + Pulse server gated on CLAUDE_MODE
install-skill-clusters.sh # Codex resolver hook, always
install-peon-ping.sh      # voice detection, gated on VOICE_MODE + macOS
install-codegraph.sh      # codegraph CLI detection + index init
install-gsd.sh            # detect-only gsd-core reference, gated on GSD_MODE
configure-opencode.sh     # OpenCode/Cursor/Codex templates, gated per-surface
verify-install.sh         # runs LAST, always, and fails the whole install if it fails

2. Shared Primitives (scripts/lib.sh)

is_dry_run

Checks TEMPERANCE_DRY_RUN=1. Every mutating primitive below checks this first and substitutes a DRY_RUN: ... print for the real action.

ensure_dir DIR

mkdir -p, or its dry-run echo.

backup_file PATH

No-ops if PATH doesn't exist yet. Otherwise copies it to $TEMPERANCE_BACKUP_DIR/<UTC timestamp>/<basename> before anything overwrites it — this is the entire rollback mechanism.

install_file SRC DEST

ensure_dir the parent, backup_file DEST, then cp SRC DEST. Every template write in the repo goes through this one function — there is no other write path.

There is no JSON-merge or partial-file-patch primitive anywhere in this codebase. Every install is a whole-file cp; anything that needs manual merging (e.g. an existing opencode.json) is intentionally left to the user with a printed instruction, never auto-merged.

3. Per-Script Internals

install-pai.sh AGENTS.md always rest gated on --with-claude

Always installs templates/AGENTS.md → $HOME/AGENTS.md, regardless of any flag. If CLAUDE_MODE=install: creates $PAI_HOME/PAI/PULSE and $PAI_HOME/hooks, installs CLAUDE.md.template and compat-server.ts, then — if bun is on PATH — manages a PID lifecycle: reads any existing compat-server.pid, kill -0-checks it's alive, kills it, then nohup bun compat-server.ts &s a fresh instance and writes the new PID. If bun is absent, the server files are installed but nothing is started.

install-peon-ping.sh gated on VOICE_MODE + macOS

Exits immediately if VOICE_MODE=skip. Exits immediately on any non-Darwin host unless VOICE_MODE=force. Otherwise checks that $PEON_SCRIPT (default $PAI_HOME/hooks/peon-ping/peon.sh) exists and is executable — if not, it prints guidance and exits 0 (never fails the install). It never writes an audio file; it only confirms the local script is reachable.

install-skill-clusters.sh resolver hook always installed

Always copies skill_cluster_resolver.mjs to $CODEX_HOME/hooks/ (Codex-facing only — Claude/OpenCode get routing docs, not a hook). Separately checks whether $AGENTS_HOME/skill-clusters already exists with a package.json; if so, prints the three verification commands (npm run health/audit-refs/tier) as a hint, but never runs them.

install-codegraph.sh detect + conditional init

If $AGENTS_HOME exists and the codegraph CLI is on PATH: if $AGENTS_HOME/.codegraph already exists, no-ops; otherwise runs codegraph init -i inside $AGENTS_HOME (or prints the dry-run equivalent). If either precondition is missing, prints a guidance line and does nothing.

install-gsd.sh detect + print only, zero writes

Gated on GSD_MODE=install (default skip). When active, detects gsd-core (open-gsd/gsd-core) two ways: the legacy path $GSD_HOME (default ~/.claude/get-shit-done — deprecated danielmiessler lineage, kept for back-compat detection only) OR a gsd binary on PATH. If either is found, it prints a pointer to docs/pai-flow.md for how gsd-core's phases map onto the PAI 7-phase flow. If neither is found, it prints that Temperance does not vendor GSD and recommends installing gsd-core via npx @opengsd/gsd-core@latest, again pointing at docs/pai-flow.md (for the recommended-default flow and its superpowers fallback) instead of a phase-mapping doc. No ensure_dir, no install_file — this script cannot mutate anything by construction, in either branch.

configure-opencode.sh per-surface gates

OpenCode (default install): writes opencode.AGENTS.md always when enabled, but only writes opencode.json.patch.json as opencode.json if no config exists yet — otherwise it prints "merge manually" and leaves the existing file untouched. Cursor (default install): always copies both Cursor templates into $CURSOR_HOME/templates as a project-local staging area — nothing is copied into an actual project automatically. Codex (default skip): writes codex.AGENTS.md only with --with-codex.

verify-install.sh runs unconditionally, can fail the install

Four independent check categories, all must pass or the whole install exits non-zero:

  1. File existence — 21 required paths (was 18 before the parallel-dispatch work added 3 more; docs/parallel-dispatch.md itself is now a retired redirect stub pointing at docs/pai-flow.md, not deleted).
  2. Shell syntax — every *.sh in repo root + scripts/ is linted with bash -n or sh -n, chosen by sniffing its shebang line.
  3. Private-path denylist — greps README, docs, scripts, templates, package, skills, and top-level docs for a small set of local-machine path prefixes (home-directory paths, this author's specific volume mount, a hashed local-agent pattern). Any match fails the build — including, deliberately, this sentence if it spelled the literal patterns out, which is why they're described rather than quoted here.
  4. Content assertions — targeted grep -q checks that specific README/credits/Cursor-rule strings exist (banner + skills.sh reference, Thoughtseed Labs + upstream credits, "OpenCode/Cursor-first" + optional-Claude/Codex language).

Deliberately not checked: exact prose wording of the ISC-28..31 or ISC-34..37 criteria, or the exact prose of docs/pai-flow.md (the current authoritative unified-flow doc) beyond its existence — content-assertion grepping is reserved for load-bearing public claims, not every doc, to avoid coupling verification to prose that will get copyedited.

4. Pulse Compatibility Server (compat-server.ts)

A ~65-line Bun HTTP server bound to 127.0.0.1:31337 (overridable via TEMPERANCE_PULSE_PORT). Two routes:

RouteBehavior
GET /healthzReturns { ok: true, compatibility: true, port }.
POST /notifyParses { message } from the body, lowercases it, and pattern-matches for phase keywords.
This is string-sniffing, not a structured protocol. Phase detection is literally message.includes("entering the " + phase + " phase") for each of the 7 phases, plus special-cased checks for "native mode" and "entering the algorithm". There is no phase field, no schema — whatever calls /notify must phrase its message exactly that way or no pack plays (the endpoint still returns 200 {ok:true, played:false}, never an error).

On a phase match, it calls Bun.spawn([peonScript, "--pack", pack, "--category", category]) where category is session.start for native/algorithm and task.acknowledge for the 7 named phases. If Bun is undefined (shouldn't happen inside a Bun server, but defensively checked) it returns {played: false, reason: "bun-runtime-required"} instead of throwing.

5. Skill Resolver Hook (skill_cluster_resolver.mjs)

Per docs/pai-flow.md, the skill-cluster resolver is the discovery / lazy-load layer that sits between "the gsd-core (or superpowers) backbone names a skill" and "a tool actually runs." Its index (~/.agents/skill-clusters/skill-index.json) classifies every skill into one of four states, and this is the context-economy mechanism that keeps the always-loaded surface small:

  • active-spoke — already symlinked into ~/.agents/skills; resolves immediately, no extra step.
  • deferred-* — not loaded by default; the indexed activate command runs on demand, only when a task genuinely needs that cluster.
  • archived — read directly from its indexed path (or restored via the indexed restore command) without permanent registration.
  • not-found — a hard error; nothing silently falls through.

The Codex-facing hook itself is a thin, non-blocking probe of that layer, not the layer's implementation: an ~18-line Node script installed only to $CODEX_HOME/hooks/. It reads $HOME/.agents/skill-clusters/skill-index.json (or $SKILL_CLUSTER_INDEX override); if the file is missing it prints {"continue": true, "temperance": "skill-index-missing", indexPath}, otherwise {"continue": true, "temperance": "skill-index-present", "skills": <count>}. It always prints "continue": true — this hook can only inform Codex that the resolver layer exists, never block it or perform the resolution itself. Claude and OpenCode get the same layer via routing docs instead of a hook (see install-skill-clusters.sh above).
