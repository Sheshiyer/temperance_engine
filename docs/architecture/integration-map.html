Integration Map Where the stitched-together systems actually connect — and where they deliberately don't

Reading this doc. Every seam below is labeled WIRED (a real code path in this repo reads/writes/calls the other system) or REFERENCE ONLY (docs point at it; nothing in this repo's code touches it). Confusing the two is the most common misunderstanding about this runtime.

1. The Backbone: 7-Phase Loop

Observe → Think → Plan → Build → Execute → Verify → Learn is the spine everything else attaches to. It is defined once, in docs/pai-flow.md, as a convention the agent follows — there is no state machine or enforcement code for it in this repo. Every other seam below is something that can happen at a specific phase, not something the loop itself triggers.

The one deliberate bridge: ParallelDispatchContext.hook.sh reads GSD's .planning/config.json (model_profile, workflow.auto_advance) read-only, purely to enrich its own advisory output. It never writes to that file, never overrides GSD's own fresh read of it, and never reads PAI's steering rules, memory, or this project's ISA.md — those three stay exactly as separate as the table above says they should.

5. Parallel-Dispatch Seam REFERENCE ONLY (hook) + WIRED (docs cross-link)

The hook only informs — it never calls either mechanism. The decision to use GSD vs. superpowers dispatch vs. plain sequential work is always made by the agent reading the doc, never automated.
