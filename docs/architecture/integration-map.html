Integration Map Where the stitched-together systems actually connect — and where they deliberately don't

Reading this doc. Every seam below is labeled WIRED (a real code path in this repo reads/writes/calls the other system) or REFERENCE ONLY (docs point at it; nothing in this repo's code touches it). Confusing the two is the most common misunderstanding about this runtime.

1. The Backbone: 7-Phase Loop

Observe → Think → Plan → Build → Execute → Verify → Learn is the spine everything else attaches to. It is defined once, in docs/pai-flow.md, as a convention the agent follows — there is no state machine or enforcement code for it in this repo (enforcement is a future sub-project). Every other seam below is something that can happen at a specific phase, not something the loop itself triggers.

Underneath that spine, docs/pai-flow.md also names gsd-core (open-gsd/gsd-core) as the recommended-default workflow backbone (Discuss → Plan → Execute → Verify → Ship, /gsd-* commands) that the 7 phases fold onto. It is recommended, not required, and still detect-only at install (--with-gsd never vendors it) — when absent, the flow degrades to the superpowers chain (brainstorming → writing-plans → subagent-driven-development → verification-before-completion → finishing-a-development-branch). See seam 3b below for the skill-cluster resolver that sits between a named skill and a tool actually running.

3b. Skill-Cluster Resolver Layer WIRED — the discovery/lazy-load layer between backbone and tool

Promoted out of the terse mention above because docs/pai-flow.md now documents it as a real architecture layer, not a footnote: it sits between “gsd-core (or any backbone) names a skill” and “a tool actually runs.” This is the context-economy mechanism — it is why the agent doesn't load all ~30+ skill clusters on every turn.

Resolution stateWhat happens
active-spokeAlready symlinked into $AGENTS_HOME/skills by tier.mjs --apply — loads immediately
deferred-*Cluster resolved on demand via its indexed activate command, only when the task genuinely needs it
archivedRead directly from the indexed path, or restored persistently via the indexed restore command
not foundHard error — no silent fallback to scanning the full skill tree

Index file: $AGENTS_HOME/skill-clusters/skill-index.json. Health is checked with npm run health, npm run audit-refs, and npm run tier inside $AGENTS_HOME/skill-clusters. The index and the four resolution states are real and consulted before any skill lookup — the only REFERENCE ONLY part left is that a hook enforcing this automatically is currently installed for Codex alone (see the table above); other surfaces follow the same discipline as a doc convention.

5. Parallel-Dispatch Seam REFERENCE ONLY (hook) + WIRED (temperance-batch is a real, runnable CLI)

This seam was simplified when docs/parallel-dispatch.md retired to a redirect stub: the Execute-phase dispatch decision is no longer a separate decision tree to go consult — it is now the Execute row + doctrine in docs/pai-flow.md itself. There is one doc to read, not two.

The hook only informs — it never calls either mechanism. What is genuinely WIRED here is package/router/dispatch-tasklist.sh, installed as the temperance-batch command: it is a real, runnable CLI that fans work out to external backends (command-code/grok/kimi), not just advisory text. It fires only in Execute (and optionally Verify) per docs/pai-flow.md doctrine — never a default for single-agent work. package/conductor/routed-execute.sh is deleted; dispatch-tasklist.sh supersedes it. The decision itself (gsd-core vs. superpowers dispatch vs. plain sequential work) is always made by the agent reading docs/pai-flow.md, never automated.
