Session Trace One concrete walkthrough, from installer flags to a finished Execute phase

Worked example. This traces a single real invocation: ./install.sh --with-claude --with-gsd --with-voice, followed by one live session that reaches the Execute-phase decision point. gsd-core is the recommended-default backbone but is never vendored by this installer — --with-gsd is detect-only. Part B traces the full-backbone path (gsd-core present); Part C's first row traces the degrade-gracefully path (gsd-core absent → superpowers chain). Every step cites the exact file/function from system-internals.html; the authoritative decision doc is pai-flow.md.

Part A — Install

  1. Flags parsed. CLAUDE_MODE=install, GSD_MODE=install, VOICE_MODE=force (force needed unless on macOS); all others stay at their defaults (OpenCode/Cursor install, Codex skip).
  2. install-pai.sh runs. $HOME/AGENTS.md written. Because CLAUDE_MODE=install: $PAI_HOME/PAI/PULSE and $PAI_HOME/hooks created, CLAUDE.md.template and compat-server.ts copied in (each preceded by a timestamped backup of anything already at that path). If bun exists, any stale compat-server.pid process is killed and a fresh server is spawned on :31337.
  3. install-skill-clusters.sh runs. skill_cluster_resolver.mjs copied to $CODEX_HOME/hooks/ unconditionally. If $AGENTS_HOME/skill-clusters/package.json exists, the three verification commands are printed as a hint.
  4. install-peon-ping.sh runs. Not skipped (voice forced). Checks $PAI_HOME/hooks/peon-ping/peon.sh is executable. If it isn't yet (peon-ping wasn't installed separately), prints guidance and exits 0 — the install continues either way.
  5. install-codegraph.sh runs. If $AGENTS_HOME exists and codegraph is on PATH, runs codegraph init -i once (skipped if .codegraph already present).
  6. install-gsd.sh runs. GSD_MODE=install, so it checks for gsd-core (open-gsd/gsd-core, installed via npx @opengsd/gsd-core@latest). On this machine it's present → prints "gsd-core detected" + a pointer to pai-flow.md's Execute row. Still detect-only: --with-gsd never vendors gsd-core itself. Zero files written. (The legacy $HOME/.claude/get-shit-done path is deprecated — install-gsd.sh no longer looks there first.)
  7. configure-opencode.sh runs. OpenCode and Cursor templates land in their default locations; Codex is skipped (not requested).
  8. verify-install.sh runs. All 21 required files present, all shell scripts lint clean, no private paths found, all content assertions pass → Temperance Engine verification passed. If any check had failed, install.sh would exit non-zero here — nothing after this point would print.
  9. Install flow complete. User is told to restart OpenCode/Cursor to reload instruction surfaces (Claude/Codex only if those optional surfaces were touched).

Part B — A Live Session Reaching Execute (gsd-core present)

  1. Session starts.setup If the user manually wired ParallelDispatchContext.hook.sh into their own ~/.claude/settings.json (it is not auto-registered), it checks .planning/ in the current project. Say this project has one, with 2 active workstreams and a config.json setting model_profile: "quality" — the hook injects a <system-reminder> noting all three (read-only) and pointing at pai-flow.md (the Execute row + doctrine, not the retired docs/parallel-dispatch.md). It never reads PAI's steering rules or writes anything back to config.json — Temperance Engine's own preference store stays ISA.md alone. Separately, the enrichment core emits a <temperance-context> block on every prompt regardless — the only always-on layer in this trace.
  2. Observe.observe Agent reads live state. If it emits "Entering the Observe phase." and Pulse is running, compat-server.ts matches phase=observe → spawns peon.sh --pack glados --category task.acknowledge.
  3. Think.think Same pattern → pack hal_2001.
  4. Plan.plan Pack jarvis-mk2. gsd-core is present (confirmed at install time), so the agent invokes /gsd-plan-phase here (hyphen form — Claude Code / gsd-core spelling; Gemini CLI would spell it /gsd:plan-phase, Codex $gsd-plan-phase). The command names the skill it needs; the skill-cluster resolver (~/.agents/skill-clusters/skill-index.json) resolves it — active-spoke skills are already symlinked and load immediately, a deferred cluster is activated on demand, an archived one is read from its indexed path, and a not-found name is a hard error. This lazy-load layer is what keeps context small while /gsd-* and superpowers skills alike stay reachable.
  5. Build.build Pack peon. gsd-core has no separate Build phase — this folds into Execute. Artifacts get created/edited via subagent-driven-development / test-driven-development.
  6. Execute — the decision point.execute Pack nier-2b. Before running the work, the agent consults pai-flow.md's Execute row + doctrine (the authoritative decision doc — docs/parallel-dispatch.md is now a retired redirect stub). Because gsd-core is present, the agent runs /gsd-execute-phase (wave-based, worktree-isolated) as the default path. If 2+ independent tasks in that execution would benefit from external backends (command-code/grok/kimi), the agent additionally fires temperance-parallel-dispatch via temperance-batch — a specialist tool layered on top of, not a replacement for, superpowers:dispatching-parallel-agents (the Claude-subagent primitive). If any task shares files/branch with another, everything runs sequentially regardless. See the table in Part C for what happens when gsd-core is absent instead.
  7. Verify.verify Pack cortana. If this session were working on Temperance Engine itself, ./verify.sh would run here as the fresh-evidence check the Completion Rule requires.
  8. Learn.learn Pack sc_kerrigan. If this is an ISA-tracked project, decisions and verification evidence get appended to its ISA.md — exactly the pattern used for ISC-28..31 in this repo's own ISA.md.

Part C — Failure Modes, Traced

Missing pieceWhat actually happens
gsd-core absent (degrade-gracefully path)At install time, install-gsd.sh finds no gsd-core (and no legacy ~/.claude/get-shit-done either) → prints the not-found branch; still detect-only, nothing is vendored. At runtime, pai-flow.md's Execute row degrades to the superpowers column, which is the flow: brainstorming → writing-plans → subagent-driven-development → verification-before-completion → finishing-a-development-branch. No /gsd-* commands are available anywhere in Part B's trace; every phase pill still fires (Pulse doesn't care which backbone is active), but Plan/Execute/Verify/Learn route through the superpowers skills instead of /gsd-plan-phase / /gsd-execute-phase / /gsd-verify-work / /gsd-ship. temperance-parallel-dispatch can still fire in Execute on its own trigger (2+ independent tasks, external backends) — it doesn't depend on gsd-core.
bun not installedPulse server files are copied but never started; any /notify POST simply has nowhere to land (connection refused). Session continues normally with no voice.
peon.sh missing or not executablecompat-server.ts still responds 200 to /notify with {played: false, reason: "bun-runtime-required"}-style fields; no sound, no error surfaced to the agent.
Skill-cluster resolver can't find a named skillActive-spoke and deferred lookups both fail against ~/.agents/skill-clusters/skill-index.json → hard error, by design; there is no silent fallback for a genuinely unknown skill name (only gsd-core-vs-superpowers is a documented degrade path).
codegraph CLI not installedStructural search guidance in docs/codegraph-routing.md is simply unusable; no fallback is installed automatically (the doc says to use direct file reads/grep instead).
